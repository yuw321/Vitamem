# Deduplication

Vitamem uses **cosine similarity** to prevent storing semantically duplicate memories. When new facts are extracted from a conversation, they are compared against all existing memories for that user before being saved.

## How It Works

For each newly extracted fact:

1. The fact is embedded into a vector
2. The vector is compared against all existing user memory vectors using cosine similarity
3. If similarity ≥ threshold (default: **0.92**), the fact is considered a duplicate and skipped
4. If similarity < threshold for all existing memories, the fact is saved

Additionally, new facts within the same batch are deduplicated against each other — so if the same conversation produces two similar facts, only the first one is kept.

## The Similarity Threshold

The default threshold of **0.92** is calibrated to:

- Catch near-identical phrasings ("Takes metformin" vs "Is on metformin")
- Allow genuinely distinct but related facts to coexist ("Takes metformin 500mg" vs "Takes metformin 1000mg")

You can adjust this threshold when initializing:

```typescript
// Lower threshold = more aggressive deduplication (fewer, broader memories)
// Higher threshold = more permissive (more memories, more specificity)
const mem = await createVitamem({
  provider: "openai",
  apiKey: process.env.OPENAI_API_KEY!,
  storage: "ephemeral",
});
// deduplication threshold is set in the embedding pipeline, default 0.92
```

## Example

Suppose the user has an existing memory: **"Has Type 2 diabetes"**.

In a new conversation they say: _"As a diabetic, I need to watch my diet carefully."_

The extraction might produce: **"Is diabetic"**

Cosine similarity between "Has Type 2 diabetes" and "Is diabetic" vectors ≈ 0.95 → **duplicate, skipped**.

Later they say: _"My A1C was 7.2 at my last checkup."_

Extraction produces: **"A1C was 7.2"**

Cosine similarity against all existing memories < 0.92 → **new fact, saved**.

## Memory Supersede

Starting with Vitamem's two-tier threshold system, facts are not simply "duplicate or new" — there is a middle ground for **updated information**.

### Two-Tier Threshold System

| Similarity Range | Classification | Action |
|-----------------|----------------|--------|
| `>= 0.92` (deduplicationThreshold) | Exact duplicate | New fact is **discarded** |
| `>= 0.75` and `< 0.92` (supersedeThreshold) | Same topic, updated value | Existing memory is **updated in-place** |
| `< 0.75` | New distinct fact | **Saved** as a new memory |

### How Supersede Works

When a new fact falls in the supersede band (between `supersedeThreshold` and `deduplicationThreshold`), Vitamem recognizes it as the same topic with an updated value and replaces the existing memory content rather than creating a conflicting duplicate.

**Example:** A user says _"My last A1C was 7.4%"_ and that fact is stored. Months later they say _"Current A1C is 6.8%"_. The cosine similarity between these two facts is ~0.88 — above the supersede threshold (0.75) but below the dedup threshold (0.92). Instead of storing both values (which would create conflicting memories), the old memory is updated to "Current A1C is 6.8%".

:::note[Source and Pinned Preservation]
When a memory is superseded, Vitamem preserves the higher-confidence source classification. A "confirmed" memory will never be downgraded to "inferred" during supersede. Similarly, pinned status is preserved — a pinned memory stays pinned after supersede.
:::

### Configuring Thresholds

Both thresholds are configurable via `VitamemConfig`:

```typescript
const mem = await createVitamem({
  provider: "openai",
  apiKey: process.env.OPENAI_API_KEY!,
  storage: "ephemeral",
  deduplicationThreshold: 0.95, // stricter duplicate detection
  supersedeThreshold: 0.80,     // narrower supersede band
});
```

:::tip
The default thresholds (0.92 / 0.75) are calibrated for health companion use cases. Adjust them based on your domain — a narrower supersede band reduces false supersedes at the cost of more conflicting memories.
:::

## Why This Matters for Health Apps

Without deduplication, users who check in regularly would accumulate hundreds of redundant memories ("Has diabetes", "Is diabetic", "Manages diabetes", ...). This degrades retrieval quality because the vector store becomes diluted with near-duplicate entries.

Vitamem's deduplication keeps the memory store lean and retrieval precise — which is especially important for health companions that may interact with a user daily over months or years.
