Embedding Pipeline

API reference for the embedding pipeline that processes threads transitioning to dormant.

Embedding Pipeline

The embedding pipeline is the core process that runs when a thread transitions to dormant. It extracts facts from the conversation, embeds them as vectors, deduplicates against existing memories, and saves the unique ones.

import { runEmbeddingPipeline } from "vitamem";

In normal usage, you do not call this function directly -- the Vitamem facade runs it automatically during triggerDormantTransition() and sweepThreads(). It is exported for advanced use cases where you need direct control over the pipeline.

runEmbeddingPipeline()

Process a dormant thread's messages into deduplicated, embedded memories.

Signature

async function runEmbeddingPipeline(
  thread: Thread,
  messages: Message[],
  llm: LLMAdapter,
  storage: StorageAdapter,
  deduplicationThreshold?: number,
  concurrency?: number,
): Promise<EmbeddingPipelineResult>;

Parameters

Parameter Type Default Description
thread Thread -- The thread that transitioned to dormant. Used for userId and threadId when saving memories.
messages Message[] -- The full conversation history to extract facts from.
llm LLMAdapter -- The LLM adapter used for extraction (extractMemories) and embedding (embed).
storage StorageAdapter -- The storage adapter for reading existing memories and saving new ones.
deduplicationThreshold number 0.92 Cosine similarity threshold for deduplication. Facts with similarity >= this value to any existing memory are considered duplicates and discarded.
concurrency number 5 Maximum number of concurrent embedding API calls. Controls how many llm.embed() calls run in parallel.

Returns

interface EmbeddingPipelineResult {
  memoriesSaved: number;    // Facts that passed deduplication and were saved
  memoriesDeduped: number;  // Facts that were discarded as duplicates
  totalExtracted: number;   // Total facts extracted by the LLM
}

The invariant memoriesSaved + memoriesDeduped === totalExtracted always holds.

Pipeline Steps

The function executes five steps in order:

  1. Extract facts -- Calls llm.extractMemories(messages) to get structured facts from the conversation. Each fact has a content string and a source ("confirmed" or "inferred").

  2. Embed facts -- Calls llm.embed(fact.content) for each extracted fact. This step respects the concurrency parameter, running at most N embedding calls in parallel using a worker pool.

  3. Load existing memories -- Calls storage.getMemories(thread.userId) to get all current memories for the user, needed for deduplication.

  4. Deduplicate -- Compares each new fact's embedding against all existing memory embeddings (and already-accepted new facts) using cosine similarity. Facts with similarity >= deduplicationThreshold to any existing memory are discarded.

  5. Save unique facts -- Calls storage.saveMemory() for each unique fact, including its embedding vector.

Example

import { runEmbeddingPipeline } from "vitamem";

const result = await runEmbeddingPipeline(
  dormantThread,
  messages,
  llmAdapter,
  storageAdapter,
  0.90, // slightly lower dedup threshold
  3,    // conservative concurrency for rate-limited APIs
);

console.log(`Extracted: ${result.totalExtracted}`);
console.log(`Saved: ${result.memoriesSaved}`);
console.log(`Deduped: ${result.memoriesDeduped}`);

Concurrency Behavior

The concurrency parameter controls how many llm.embed() calls execute simultaneously. This is implemented using a worker pool pattern where N workers each process one item at a time from a shared queue.

concurrency = 3, facts = 8

Worker 1: [fact1] [fact4] [fact7]
Worker 2: [fact2] [fact5] [fact8]
Worker 3: [fact3] [fact6]

Choosing a Concurrency Value

Scenario Recommended value
OpenAI (paid tier) 5 (default)
OpenAI (free tier) 2-3
Anthropic + OpenAI embeddings 5
Ollama (local) 10+ (limited by hardware, not rate limits)
Custom embedding service Depends on your service's capacity

The facade passes config.embeddingConcurrency (default: 5) to this function. Set it in your createVitamem config:

const mem = await createVitamem({
  provider: "openai",
  apiKey: process.env.OPENAI_API_KEY,
  storage: "supabase",
  supabaseUrl: process.env.SUPABASE_URL,
  supabaseKey: process.env.SUPABASE_KEY,
  embeddingConcurrency: 3, // conservative
});

EmbeddingPipelineResult

The result type returned by runEmbeddingPipeline().

interface EmbeddingPipelineResult {
  memoriesSaved: number;
  memoriesDeduped: number;
  totalExtracted: number;
}
Field Type Description
memoriesSaved number Number of new unique memories persisted to storage
memoriesDeduped number Number of extracted facts discarded as duplicates
totalExtracted number Total number of facts the LLM extracted from the conversation

When the Pipeline Runs

In the standard Vitamem facade, the pipeline runs in two scenarios:

  1. triggerDormantTransition(threadId) -- Immediately transitions the thread to dormant and runs the pipeline.

  2. sweepThreads() -- For each cooling thread that has exceeded the cooling timeout, transitions to dormant and runs the pipeline.

Both paths call runEmbeddingPipeline() with a deduplication threshold of 0.92 and the configured embeddingConcurrency.

The pipeline is idempotent with respect to deduplication. If you accidentally run it twice on the same thread, the second run will extract the same facts but they will all be deduplicated against the memories saved by the first run.
